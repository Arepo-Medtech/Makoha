# P2.1 Increment 1 — Authoritative Evaluation Staging Preflight

**Baseline:** `Arepo-Medtech/Makoha main @ 4920d0c`  
**Date:** 21 July 2026  
**Status:** REMEDIATION PATCH PREPARED/LOCALLY VERIFIED — MERGE AND ENVIRONMENT PROOF REQUIRED; LIVE GENERATION STILL STOPPED

## Decision boundary

Do not dispatch `eval-live-staging` yet.

The current workflow can invoke the CLI with `--mode live` without proving that either generation adapter actually entered its live path. A live-labelled report and recorded fixture must never be accepted when the underlying adapter used its deterministic mock.

## Findings

### 1. Workflow secrets are not injected

`.github/workflows/staging-eval.yml` sets:

- `HEYDOC_MODE_DEFAULT=staging`; and
- `HEYDOC_LLM_LIVE=1`.

It comments that Claude and MedGemma credentials are supplied by the staging environment, but does not reference those secrets in the step environment. GitHub Actions requires available secrets to be explicitly passed as inputs or environment variables. Environment attachment controls access; it does not automatically create arbitrary process environment variables.

**Status:** RED.

### 2. MedGemma live mode is not enabled

The MedGemma adapter requires all of:

- `HEYDOC_MEDGEMMA_LIVE=1`;
- `HEYDOC_MEDGEMMA_ENDPOINT`; and
- a resolvable `HEYDOC_MEDGEMMA_KEY_REF`, defaulting to `env:HEYDOC_MEDGEMMA_KEY`.

The workflow declares none of these. MedGemma therefore falls back to its deterministic mock unless the runner is independently and unusually pre-populated.

**Status:** RED.

### 3. Eval mode does not prove adapter mode

`scripts/eval-run.mjs` stamps the report's `mode` from the CLI argument. It does not require every generation receipt to record `audit.mode=live`, nor does the EvalRunReport contain a mechanical backend-execution proof that rejects mock generation during an authoritative run.

Consequently, `--mode live` describes the fixture recorder, not necessarily the model adapter.

**Status:** RED — safety-critical provenance gap.

### 4. Default case selection is lexicographic, not representative

The manual workflow defaults to `--limit 45`. The CLI sorts case directories and takes the first 45. At the current corpus this selects only `SPEC-CARD-*` cases. It spans tiers T1–T5 but is not a cross-specialty certification set.

An initial smoke or calibration run may use a small tranche, but it must be labelled non-certifying. An authoritative gate must use either the full attested corpus or a pre-approved, manifest-pinned stratified sample with minimum tier, specialty, complexity and long-list coverage.

**Status:** RED — sample-selection bias.

## Required remediation

### A. Workflow wiring

Explicitly map protected staging secrets/variables into the live step, including at minimum:

- `ANTHROPIC_API_KEY` or an approved `HEYDOC_LLM_KEY_REF` target;
- `HEYDOC_MEDGEMMA_LIVE=1`;
- `HEYDOC_MEDGEMMA_ENDPOINT`;
- `HEYDOC_MEDGEMMA_KEY` or an approved key reference;
- any endpoint-specific model and timeout settings approved for staging.

Do not print values. Preserve required-reviewer protection on the `staging` environment.

### B. Fail-closed live-backend proof

Before any case generation, the CLI must verify that every requested backend is live-enabled. During the run it must capture and validate per-turn adapter audit evidence. A live run must fail if any generation or scoring-judge call is mock, missing, truncated or not receipt-backed.

The report schema should include a backend-execution proof or equivalent aggregate derived from immutable per-turn receipts. CLI `mode=live` alone is insufficient.

### C. Contract tests

Add tests proving that:

1. missing Claude credentials refuse an authoritative live run before generation;
2. missing MedGemma flag, endpoint or key refuses before generation;
3. a mock adapter result cannot be recorded in a live report;
4. one live and one mock backend cannot satisfy the two-backend gate;
5. the workflow explicitly references the required protected values;
6. replay remains deterministic and secret-free.

### D. Controlled evaluation set

Create a versioned evaluation manifest. Recommended sequence:

1. **Connectivity smoke:** 2–5 synthetic cases, non-certifying.
2. **Calibration run:** a stratified cross-tier/cross-specialty tranche, non-certifying.
3. **Authoritative run:** all 707 attested conforming cases, or a clinician/statistician-approved and manifest-pinned stratified design. Include an approved sample from the 343 long-list cases for positional stability.

Do not let directory sort order define the clinical test population.
